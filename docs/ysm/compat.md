# YSM 兼容性

对于该模组各种方面上兼容性的期待……我的建议是不要抱有太大期待（

## 设备兼容性

由于该模组在2.2.2版本中开始采用全新的技术实现原先简单的模型更换功能，该模组对于游玩设备的兼容性说是腰斩也不为过。

以下是设备兼容性列表：

| x86设备概况/模组版本 | 1.1.5 | 2.2.2 - 2.5.1 | 2.5.2 - 2.6.5 |
| :---------------: | :---: | :-----------: | :------------: |
| Windows x86_64 | ✅ | ✅ | ✅ |
| macOS Intel | ✅ | ❌ | ❌ |
| Linux(glibc) x86_64 | ✅ | ⚠️(仅服务端) | ✅ |
| FreeBSD x86_64 | 未知 | ❌(在Linuxulator中运行失败) | ❌(在Linuxulator中运行失败) |

| Arm设备概况/模组版本 | 1.1.5 | 2.2.2 - 2.5.1 | 2.5.2 - 2.6.5 |
| :---------------: | :---: | :-----------: | :------------: |
| Windows on Arm | 未知 | ❌ | ❌ |
| macOS M-Series | ✅ | ❌ | ❌ |
| Linux Arm | 未知 | ❌ | ❌ |
| 安卓 | ✅ | ❌ | ✅ |
| iOS | 未知 | ❌ | ❌ |

至于其他的RISC-V, 龙芯用户嘛……（

## 模组兼容性

该模组在2.2.2之后就因为高强度的对模组本体和模型的保护，导致第三方团队想要插手模组适配这件事从理论上变得相当困难。

目前为止，能获得深层次适配的模组基本均为YSM团队自己发力出来的成果。（如果放出来一个独立的库或者api,会不会让这件事变得更简单呢？）

模组兼容性列表请访问[MC百科介绍页面](https://www.mcmod.cn/class/8616.html)进行了解，我懒得把这玩意二次整理成列表了（

## 服务端软件生态兼容性

很不好（

目前官方可用的服务端实现只能是他们自己的模组扔到对应的模组端上，并且对一些插件和模组的混合服务端软件支持也算得上比较地狱。

你问我为什么没算上[Freesia](https://github.com/YesSteveModel/Freesia)？这玩意已经没有官方支持和维护了，个人认为官方事实上并不太关心这类方案的发展，并且从提交来看官方也没有直接或间接的参与进这个项目的维护当中，可以说这个项目实际上已经处于死亡状态了。

在服务端侧，存在通过Velocity网络转发方案实现的[FreesiaII](https://github.com/NguyenDevs/FreesiaII)和[FreesiaIII](https://github.com/MMeowRealms/FreesiaIII),和通过已有反编译项目OpenYSM二次开发得来的[Noir](https://github.com/EilsapMC/Noir)。均为第三方项目。

::: info
虽然FreesiaII和FreesiaIII名字相近，但并非同一项目。FreesiaII是YSM仓库的Freesia的直接衍生的接续版本，更适合用于大型网络与商业化服务器，也做了对更多服务端软件平台的支持；而FreesiaIII前身FreesiaV2是一个专注改进Freesia技术缺陷的接续版本，III作为v3延续了这一理念。
:::
